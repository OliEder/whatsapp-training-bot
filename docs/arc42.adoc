== 1. Einführung und Ziele

=== 1.1 Aufgabenstellung

Ein Jugend-Basketball-Trainer betreut mehrere Teams (U8, U10, ...). Vor jedem Training und Spiel müssen manuell WhatsApp-Umfragen erstellt werden, um die Teilnahme abzufragen. Dieser Bot automatisiert genau das:

* Liest Kalender-Feeds (ICS/iCal) für Trainings und Spiele
* Sendet automatisch WhatsApp-Polls zum konfigurierten Zeitpunkt
* Erstellt Gruppenevents für Spielankündigungen
* Exportiert Teilnahmelisten als CSV

=== 1.2 Qualitätsziele

[cols="1,3,3"]
|===
|Priorität |Qualitätsziel |Szenario

|1 |*Zuverlässigkeit* |Polls werden genau einmal gesendet, auch bei Neustarts
|2 |*Einfache Bedienung* |Einrichtung per Web-UI, kein Coding nötig
|3 |*Datenpersistenz* |WhatsApp-Session und Konfiguration überleben HA-Neustarts
|4 |*Erweiterbarkeit* |REST-API für spätere Trainer-App-Integration
|===

=== 1.3 Stakeholder

[cols="1,2,3"]
|===
|Rolle |Name/Beschreibung |Erwartungen

|Trainer |Primärnutzer |Polls automatisch, ohne manuellen Aufwand
|Eltern |WhatsApp-Gruppenempfänger |Klare, rechtzeitige Umfragen
|Entwickler |Zukünftige Trainer-App |Saubere REST-API
|===

== 2. Randbedingungen

=== 2.1 Technische Randbedingungen

[cols="1,3"]
|===
|Randbedingung |Begründung

|WhatsApp (kein Telegram) |Alle Eltern-Gruppen sind dort, kein Wechsel möglich
|Home Assistant Add-on |Läuft 24/7 ohne eigenen Server, kostenlos
|Kein WhatsApp Business API |Keine laufenden Kosten, inoffiziell via Browser-Automation
|Node.js 20 |Kompatibel mit whatsapp-web.js
|===

=== 2.2 Organisatorische Randbedingungen

* Kein Cloud-Dienst (Datenschutz, keine laufenden Kosten)
* Bedienung durch nicht-technische Nutzer
* Kalender werden in Apple iCal/iCloud gepflegt

== 4. Lösungsstrategie

=== 4.1 Architekturentscheidungen

[cols="1,2,2"]
|===
|Entscheidung |Gewählte Option |Begründung

|WhatsApp-Client |whatsapp-web.js (Browser-Automation) |Kein Business API nötig, keine Kosten
|Datenhaltung |SQLite (better-sqlite3) |Kein DB-Server, einfach, ausreichend für die Last
|Scheduler |node-cron (täglich 08:00) |Einfach, kein Echtzeit-Polling nötig
|Deployment |Docker / HA Add-on |Läuft 24/7, persistent, keine Cloud
|Frontend |Vanilla HTML/JS |Kein Framework-Overhead, einfach wartbar
|===

=== 4.2 Duplicate-Send-Prävention

Jede gesendete Aktion wird in `sent_polls` protokolliert. Vor jedem Senden prüft `alreadySent()`, ob der Eintrag existiert:

[source,sql]
----
SELECT id FROM sent_polls
WHERE team_id = ?
  AND gruppe_whatsapp_id = ?
  AND typ = ?          -- 'training' | 'spiel' | 'event'
  AND ereignis_datum = ?
----

Damit sind Neustarts und doppelte Cron-Runs sicher.

=== 5.2 Wichtige Bausteine

==== db.js — Datenbankschicht

Initialisiert das SQLite-Schema und stellt eine Singleton-Datenbankinstanz bereit.

Exports: `initDb()`, `getDb()`

==== ical.js — Kalender-Parsing

[cols="1,3"]
|===
|Funktion |Beschreibung

|`fetchIcs(url)` |Lädt ICS-Feed, normalisiert `webcal://` → `https://`
|`parseEvents(ics)` |Parst VEVENT-Einträge inkl. RRULE
|`filterEvents(events, filter)` |Case-insensitive SUMMARY-Textfilter
|`getUpcomingEvents(events, tage)` |Gibt Events zurück, die in genau N Tagen stattfinden
|===

==== scheduler.js — Zeitplanung

Läuft täglich um 08:00 via node-cron. Pro Team und Feed:

1. Training-Feed: Poll N Tage vorher an Gruppen mit Rolle `training`
2. Spiele-Feed: Event 14 Tage vorher (`allgemein`), Poll N Tage vorher (`spiel-orga`), CSV 1 Tag vorher

==== whatsapp.js — WhatsApp-Client

Verwaltet den Puppeteer-gesteuerten WhatsApp-Web-Browser. Session wird via `LocalAuth` in `/data/wwebjs_auth` persistiert.

==== polls.js / events.js / csv.js

Fachlogik für die drei Ausgabe-Typen:

* **polls.js**: Formatiert Datum auf Deutsch, sendet Multi-Select (Training) oder Single-Select (Spiel)
* **events.js**: Erstellt WhatsApp-Gruppenevent mit Titel, Ort und Zeitangabe
* **csv.js**: Generiert CSV aus Umfrageergebnissen, filtert "Mein Kind ist nicht in der Liste"

== 6. Laufzeitsicht

=== 6.1 Szenario: Training-Poll senden

[source,mermaid]
----
sequenceDiagram
    participant Cron as node-cron (08:00)
    participant Sched as scheduler.js
    participant iCal as ical.js
    participant DB as SQLite
    participant Polls as polls.js
    participant WA as whatsapp.js

    Cron->>Sched: runDailyCheck()
    Sched->>DB: SELECT * FROM teams
    loop Pro Team
        Sched->>iCal: fetchIcs(training_feed_url)
        iCal-->>Sched: ICS-String
        Sched->>iCal: parseEvents() + filterEvents()
        iCal-->>Sched: Events[]
        loop Pro Event
            Sched->>Sched: shouldSendInDays(event, tage_vor_training)?
            alt Ja
                Sched->>DB: alreadySent(team, gruppe, 'training', datum)?
                alt Nein
                    Sched->>Polls: sendTrainingPoll(team, gruppeId, start)
                    Polls->>WA: sendPoll(gruppeId, frage, optionen, multiSelect=true)
                    WA-->>Polls: messageId
                    Polls->>DB: INSERT INTO sent_polls
                end
            end
        end
    end
----

=== 8.2 Fehlerbehandlung

[cols="1,2,2"]
|===
|Fehlerfall |Behandlung |Konsequenz

|ICS-Feed nicht erreichbar |`try/catch` im Scheduler, `console.error` |Nächster Cron-Lauf versucht es erneut
|WhatsApp nicht verbunden |HTTP 503 von `/api/whatsapp/gruppen` |Web-UI zeigt Status, kein Poll-Absturz
|Poll bereits gesendet |`alreadySent()` bricht ab |Kein doppelter Versand
|Ungültige Team-ID |HTTP 404 in API-Routen |Klare Fehlermeldung
|===

=== 8.3 Sicherheit

* WhatsApp-Session liegt im lokalen `/data`-Volume (kein Cloud-Upload)
* Web-UI nur über HA Ingress erreichbar (kein direkter Internetzugang)
* Keine Passwörter gespeichert — Authentifizierung via QR-Code

== 9. Architekturentscheidungen

=== ADR-001: whatsapp-web.js statt WhatsApp Business API

*Status:* Akzeptiert

*Kontext:* Für automatische WhatsApp-Nachrichten gibt es zwei Wege: die offizielle Business API (kostenpflichtig, erfordert verifiziertes Business-Konto) oder Browser-Automation via whatsapp-web.js.

*Entscheidung:* whatsapp-web.js mit Puppeteer.

*Konsequenzen:*
* (+) Kostenlos, keine monatlichen Gebühren
* (+) Unterstützt native WhatsApp-Polls und Group-Events
* (-) Inoffizielle API, WhatsApp könnte Änderungen vornehmen
* (-) Benötigt Chromium im Container (~300 MB)

=== ADR-002: SQLite statt PostgreSQL/MySQL

*Status:* Akzeptiert

*Kontext:* Die Last ist minimal (ein Schreibvorgang pro Tag pro Team), aber Persistenz über Neustarts ist kritisch.

*Entscheidung:* SQLite via better-sqlite3.

*Konsequenzen:*
* (+) Kein eigener DB-Server, einfaches Setup
* (+) Single-File-Datenbank leicht zu sichern
* (-) Kein Multi-Process-Zugriff (nicht relevant hier)

=== ADR-003: Täglicher Cron statt Event-getriebenem Scheduler

*Status:* Akzeptiert

*Kontext:* Polls müssen X Tage vor einem Termin versendet werden.

*Entscheidung:* node-cron läuft täglich um 08:00 und prüft alle Teams.

*Konsequenzen:*
* (+) Einfach, keine komplexe Event-Queue
* (+) `alreadySent()` verhindert doppelten Versand bei Mehrfachläufen
* (-) Max. 24h Verzögerung bei Konfigurationsänderungen (akzeptabel)

== 12. Glossar
